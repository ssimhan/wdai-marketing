# PRD: Phase 3, Block B — Promo Planner Integration

**Date:** 2026-04-14
**Status:** Approved — ready to plan
**Branch:** phase-3-block-b

---

## Goal

Extend the Luma calendar sync so that each event in `vault/content-calendar.md` is automatically enriched with:
- A **DRI** (directly responsible individual) inferred from event type
- A **channel plan** (promo timeline) generated from per-event-type rules

Manual edits to either field must survive daily re-syncs.

---

## Scope — What We Are NOT Building

- No web UI or interactive CLI prompts
- No notification system or Slack posting
- No database or external state store
- No change to the output file path (`vault/content-calendar.md`)
- Not filling in the actual promo rule content (user will do this via Slack MCP)

---

## Architecture

### New files

| File | Purpose |
|------|---------|
| `tools/calendar/promo-rules.yaml` | Per-event-type rules: DRI + channel timeline. **Content TBD by user.** |
| `tools/calendar/overrides.yaml` | Per-event manual overrides, keyed by `luma_id`. Human-edited. |
| `tools/calendar/rules-loader.ts` | Reads and validates both YAML files; returns typed rule + override maps |

### Updated files

| File | Change |
|------|--------|
| `tools/calendar/mapper.ts` | Accept `PromoRules` and `OverridesMap` as inputs; apply priority logic |
| `tools/calendar/sync.ts` | Load rules + overrides; pass into mapper |
| `tools/calendar/types.ts` | Add `PromoRules`, `ChannelAction`, `OverridesMap` types |

### Data flow

```
sync.ts
  ├── fetchAllEvents()      → LumaEvent[]
  ├── loadPromoRules()      → PromoRules       (from promo-rules.yaml)
  ├── loadOverrides()       → OverridesMap     (from overrides.yaml)
  ├── events.map(e =>
  │     mapLumaEvent(e, rules, overrides))  → CalendarEntry[]
  └── renderCalendar(entries) → vault/content-calendar.md
```

### Priority logic (in mapper)

```
override[luma_id].dri          → wins (manual edit)
  └── rules[event_type].dri    → wins (config rule)
    └── ''                     → default (blank)

override[luma_id].channel_plan → wins (manual edit)
  └── rules[event_type].channel_plan (rendered) → wins (config rule)
    └── '*(Generated by promo planner — Phase 4)*' → default placeholder
```

### `overrides.yaml`

```yaml

# Example:
# overrides:
#   evt-mock-001:
#     dri: "Helen"
#     channel_plan: "Custom plan for this specific event"
```

## Success Criteria

- [ ] `npm run calendar:sync:mock` writes `vault/content-calendar.md` with DRI and channel plan populated from `promo-rules.yaml`
- [ ] An entry in `overrides.yaml` for a specific `luma_id` overrides the rule-generated values
- [ ] Running sync twice with the same rules + overrides produces identical output (idempotent)
- [ ] Events with no matching rule get empty DRI and the existing placeholder channel plan text
- [ ] Missing or empty `promo-rules.yaml` / `overrides.yaml` do not crash the sync (graceful fallback to defaults)

---

## Edge Cases & Failure Modes

| Case | Behavior |
|------|----------|
| `promo-rules.yaml` missing | Log warning, proceed with all defaults |
| `overrides.yaml` missing | Log warning, proceed with no overrides |
| Event type not in rules | Fall back to empty DRI, placeholder channel plan |
| `luma_id` in overrides but not in events | Silently ignore (override is unused) |
| Malformed YAML | Throw with clear error message before any writes |
